# Plan: Diagnose and fix Grok 403 with healthy anti-ban Docker environment

## What I found

- Current Grok Web route sends SSO-cookie requests directly from the Node server to `https://grok.com/rest/app-chat/conversations/new` using `grokTemporaryHeaders()` in `server/index.ts`.
- The “temporary account check” path already has proxy selection work in progress: `TemporaryAccountCheckOptions.proxy`, `temporaryAccountCheckProxyFromBody()`, and UI controls for 系统代理/直连. This should help the health check use a “healthy environment”.
- The app currently has **no Dockerfile / docker-compose** and no anti-ban Docker integration in the repo. A search found only a future TODO in docs.
- For Grok 403/HTML Cloudflare pages, existing code reports generic Cookie/proxy/browser-check messages, but it does not capture enough diagnostics to prove whether the failure is: bad/expired SSO, missing `cf_clearance`, wrong egress IP/environment, Cloudflare challenge page, or request fingerprint mismatch.

## Implementation plan

1. Preserve existing user changes
   - Keep the current UI/style edits and proxy-check changes intact.
   - Only make targeted changes around Grok diagnostics / Docker proxy integration.

2. Add Grok 403 diagnostics
   - Improve Grok failure classification in `server/index.ts` for both quota checks and chat routing.
   - Detect and report separately:
     - expired/invalid SSO/session,
     - Cloudflare / browser verification HTML,
     - missing browser cookies such as `cf_clearance`,
     - upstream `403` with non-HTML JSON/text body,
     - network/proxy failure.
   - Include safe, masked diagnostic details in request logs: status, content-type, response preview, whether SSO and Cloudflare browser cookies were present, and proxy mode/source.

3. Make health checks use the anti-ban environment explicitly
   - Keep “系统代理” as default for temporary account checks.
   - Add/import support for a custom proxy URL if needed by the anti-ban Docker service, so checks can target that environment instead of only system/direct.
   - Thread the same proxy setting to single-account refresh and import auto-check, so newly imported Grok accounts are checked in the selected environment.

4. Add Docker deployment artifacts if missing
   - Add a minimal `Dockerfile` and `docker-compose.yml` for SamAPI itself.
   - Add env placeholders for external anti-ban/proxy service, e.g. `SAMAPI_PROXY_URL` / `HTTPS_PROXY`, without baking credentials into the repo.
   - Document that the anti-ban container/service must expose an HTTP/SOCKS proxy and SamAPI should use it via system/custom proxy mode.

5. Verification
   - Run `pnpm build`.
   - If build succeeds, follow saved project instruction: `brew services stop samapi` then `brew services start samapi`.
   - Report exact build/restart result and any remaining Docker/runtime issue.
